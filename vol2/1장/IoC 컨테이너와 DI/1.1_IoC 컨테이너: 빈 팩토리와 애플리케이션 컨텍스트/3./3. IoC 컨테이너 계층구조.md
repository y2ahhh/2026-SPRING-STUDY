## IoC 컨테이너 계층구조
빈을 담아둘 IoC 컨테이너는 애플리케이션마다 하나씩이면 충분하다. 빈의 개수가 많아져거 설정파일이 커지는게 문제라면 파일을 여러 개로 쪼개서 만들고 하나의 애플리케이션 컨텍스트가 여러 개의 설정파일을 사용하게 하면 그만이다. 하지만 트리 모양의 계층구조를 만들 때 한 개 이상의 IoC 컨테이너를 만들어두고 사용해야 한다.

<br>

### 부모 컨텍스트를 이용한 계층구조 효과

각자 독립적으로 자신이 관리하는 빈을 갖고 있긴 하지만 DI를 위해 빈을 찾을 때는 부모 애플리케이션 컨텍스트의 빈까지 모두 검색한다. 먼저 자신이 관리하는 빈 중에서 필요한 빈을 찾아보고, 없으면 부모 컨텍스트에게 빈을 찾아달라고 요청한다. 계층구조를 따라서 가장 위에 존재하는 루트 컨텍스트까지 요청이 전달된다. 하지만 부모 컨텍스트에게 빈 검색을 요청하지 자식 컨텍스트에게는 요청하지 않는다.

<br>

여러 애플리케이션 컨텍스트가 공유하는 설정을 만들기 위해서 계층구조를 이용한다. 애플리케이션 안에 성격이 다른 설정을 분리해서 두 개 이상의 컨텍스트를 구성하면서 각 컨텍스트가 공유하고 싶은게 있을 때 계층구조를 이용한다.

<br>

### 컨텍스트 계층구조 테스트
**부모 설정 클래스**
```
import org.springframework.context.annotation.Bean;
import org.springframework.context.annotation.Configuration;

// 부모 컨텍스트 설정
@Configuration
public class ParentConfig {
    @Bean
    public StringPrinter printer() {
        return new StringPrinter(); // 부모에 printer 등록
    }

    @Bean
    public Hello hello(StringPrinter printer) {
        Hello hello = new Hello();
        hello.setName("Parent");   // 부모의 hello는 이름을 "Parent"로 설정
        hello.setPrinter(printer);
        return hello;
    }
}
```
**자식 설정 클래스**
```
import org.springframework.context.annotation.Bean;
import org.springframework.context.annotation.Configuration;

// 자식 컨텍스트 설정
@Configuration
public class ChildConfig {
    
    @Bean
    // 부모가 가진 printer 빈을 파라미터로 주입받음 (계층형 검색 활용)
    public Hello hello(StringPrinter printer) {
        Hello hello = new Hello();
        hello.setName("Child");    // 자식의 hello는 이름을 "Child"로 오버라이딩
        hello.setPrinter(printer);
        return hello;
    }
}
```
**테스트**
```
import org.junit.jupiter.api.DisplayName;
import org.junit.jupiter.api.Test;
import org.springframework.context.annotation.AnnotationConfigApplicationContext;
import static org.assertj.core.api.Assertions.assertThat;
import static org.assertj.core.api.Assertions.assertThatThrownBy;

class ContextHierarchyTest {

    @Test
    void testContextHierarchy() {
        // 1. 부모 컨텍스트 생성 및 초기화
        AnnotationConfigApplicationContext parentContext = new AnnotationConfigApplicationContext(ParentConfig.class);

        // 2. 자식 컨텍스트 생성 (아직 설정을 등록하지 않음)
        AnnotationConfigApplicationContext childContext = new AnnotationConfigApplicationContext();
        
        // 3. 자식 컨텍스트에게 부모 컨텍스트를 지정하여 계층구조 형성
        childContext.setParent(parentContext);
        
        // 4. 자식 컨텍스트에 설정 파일 등록 후 리프레시(갱신)
        childContext.register(ChildConfig.class);
        childContext.refresh();

        // 검증 1: 자식에게 "hello"를 요청하면 자식 설정(Child)이 부모를 오버라이딩한다.
        Hello childHello = childContext.getBean("hello", Hello.class);
        assertThat(childHello.sayHello()).isEqualTo("Hello Child"); 
        // 결과: 성공 (부모의 "Parent"가 아니라 자식의 "Child"가 나옴)

        // 검증 2: 자식 컨텍스트에는 "printer" 빈이 없지만, 부모에게서 찾아와 사용할 수 있다.
        StringPrinter parentPrinter = parentContext.getBean(StringPrinter.class);
        assertThat(childHello.getPrinter()).isSameAs(parentPrinter);
        // 결과: 성공 (자식이 부모의 printer 인스턴스를 공유해서 주입받음)

        // 검증 3: 부모 컨텍스트에게 "hello"를 요청하면 부모 자신의 빈이 나온다. (자식을 모름)
        Hello parentHello = parentContext.getBean("hello", Hello.class);
        assertThat(parentHello.sayHello()).isEqualTo("Hello Parent"); // 결과: 성공

        // 검증 4: 만약 자식에게만 있는 빈이 있다면, 부모는 그 빈을 찾을 수 없다. (상향식 검색 불가)
        // 만약 자식 컨텍스트에만 'childOnlyBean'이 있다고 가정하면, 
        // parentContext.getBean("childOnlyBean")은 NoSuchBeanDefinitionException 에러를 던진다.
    }
}
```
