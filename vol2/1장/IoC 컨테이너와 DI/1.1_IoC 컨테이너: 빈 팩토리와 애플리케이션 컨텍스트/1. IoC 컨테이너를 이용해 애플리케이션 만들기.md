## IoC 컨테이너를 이용해 애플리케이션 만들기

### IoC 컨테이너

>**IoC란?**  
**Inversion of Contol**로 **제어의 역전**이라고 한다. 스프링 애플리케이션에서는 오브젝트의 생성과 관계설정, 사용, 제거 등의 작업을 애플리케이션 코드 대신 독립된 컨테이너가 담당한다. 이를 컨테이너가 코드대신 오브젝트에 대한 제어권을 갖고 있다. 그래서 스프링 컨테이너를 IoC컨테이너라고도 한다.

<br>

`IoC를 담당하는 컨테이너 == 빈 팩토리 || 애플리케이션 컨텍스트`

<br>

프로그램이 실행될 때, 필요한 객체들을 알아서 만들고 서로 연결해주는 역할을 하기 때문에, 그 컨테이너를 **빈 팩토리**(객체 공장)라고 한다.
DI를 위한 빈 팩토리에 엔터프라이즈 애플리케이션을 개발하는 데 필요한 여러가지 컨테이너 기능을 추가한 것을 **애플리케이션 컨텍스트**라고 부른다.  
애플리케이션 컨텍스트는 빈 팩토리를 상속받았기 떄문에 자체로 IoC와 DI를 위한 빈 팩토리이면서 그 이상의 기능을 가졌다고 보면 된다.  
스프링의 빈 팩토리와 애플리케이션 컨텍스트는 각각 기능을 대표하는 `BeanFactory`와 `ApplicationContext`라는 두 개의 인터페이스로 정의되어 있다.

<br>

### 애플리케이션 컨텍스트
`ApplicationContext 인터페이스`는 `BeanFactory 인터페이스`를 상속한 서브 인터페이스이다. `ApplicationContext 인터페이스`의 정의를 보면 `BeanFactory`로 끝나는 두 개의 인터페이스를 상속받고 있는데 이 두 개의 인터페이스는 `BeanFactory 인터페이스`의 서브 인터페이스이다. 따라서 `ApplicationContext`는 `BeanFactory 인터페이스`를 상속받고 있는 셈이다.

<br>

가장 간단하게 IoC 컨테이너를 만드는 방법은 다음과 같이 ApplicationContext 구현 클래스의 인스턴스를 만드는 것이다.
```
StaticApplicationContext ac = new StaticApplicationContext();
```
`ApplicationContext 인터페이스`를 구현한 클래스인 `StaticApplicationContext`의 오브젝트를 만들어 IoC 컨테이너가 하나 준비됐다. 하지만 아직 아무런 작동을 하지 않는 빈 컨테이너이다. 이 컨테이너가 본격적으로 IoC컨테이너로서 작동하려면 POJO 클래스와 설정 메타정보가 필요하다.

>POJO 클래스란?  
**Plain Old Java Object**의 약자로 **기본적이고 오래된 방식의 단순한 자바 객체**라는 뜻이다.

### POJO 클래스
`Hello`라는 클래스와 메시지를 받아서 이를 출력하는 `Printer 인터페이스`를 사용하고, 이 `Printer 인터페이스`를 구현한 `StringPrinter`를 구현한다.
```
public class Hello {
  String name;
  Printer printer;

  public String sayHello() {
    return "Hello " + name; 
  }

  public void print() {
    this.printer.print(sayHello());
  }

  public void setName(String name) {
    this.name = name;
  }

  public void setPrinter(Printer printer) {
    this.printer = printer;
  }
}
```

여기서 `Hello 클래스`만 봤을 때 `Printer`라는 인터페이스에만 의존한다. 실제로 헌타임 시에 어떤 구체적인 클래스의 오브젝트를 사용하게 될지는 알지도 못하고, 사실 관심도 없다.  
`Printer 인터페이스`를 구현한 클래스는 얼마든지 만들 수 있다. 이번에는 `StringBuffer`에 메시지를 넣는 방식으로 메시지를 출력하는 `StringPrinter`를 만들어보자.
```
// StringPrinter
public class StringPrinter implements Printer {

  private StringBuffer sb = new StringBuffer();

  // Printer 인터페이스의 Method(내장 버퍼에 메시지를 추가해준다.)
  public void print(String message) {
    this.sb.append(message);
  }

  // 내장 버퍼에 추가해둔 메시지를 String으로 가져온다.
  public String toString() {
    return this.sb.toString();
  }
}

// ConsolePrinter
public class ConsolePrinter implements Printer {

  public void print(String message) {
    System.out.println(message);
  }
}
```

각자 기능에 충실하게 독립적으로 설계된 POJO 클래스를 만들고, 결합도가 낮고 유연한 관계를 가질 수 있도록 인터페이스를 이용해 연결해주는 것까지가 IoC 컨테이너가 사용할 POJO를 준비하는 첫 단계이다.

<br>

두 번째로 필요한 것은 POJO클래스들 중에 애플리케이션에서 사용할 것을 선정하고 이를 IoC컨테이너가 제어할 수 있도록 적절한 메타정보를 만들어 제공하는 작업인 **설정 메타정보**이다.

### 설정 메타정보
IoC 컨테이너의 가장 기초적인 역할은 오브젝트를 생성하고 이를 관리하는 것이다. 스프링 컨테이너가 관리하는 오브젝트는 **빈**(Bean)이라고 부르는데 IoC 컨테이너가 필요로 하는 설정 메타정보는 빈을 어떻게 만들고 어떻게 동작하게 할 것인가에 관한 정보이다.

<br>

스프링의 설정 메타정보는 `BeanDefinition 인터페이스`로 표현되는 순수한 추상 정보이다. 애플리케이션 컨텍스트는 이 `BeanDefinition`으로 만들어진 메타정보를 담은 오브젝트를 사용해 IoC와 DI작업을 수행한다. `BeanDefinition 오브젝트`로 변환해주는 `BeanDefinitionReader 인터페이스`가 있으면 스프링의 설정 메타정보는 어떤 형식으로든 작성할 수 있다.

**빈 메타정보**
- 빈 아이디, 이름, 별칭:
  - 빈 오브젝트로 구분할 수 있는 식별자
- 클래스 또는 클래스 이름:
  - 빈으로 만들 POJO 클래스 또는 서비스 클래스 정보
- 스코프:
  - 싱글톤, 프로토타입과 같은 빈의 생성 방식과 존재 범위
- 프로퍼티 값 또는 참조:
  - DI에 사용할 프로퍼티 이름과 값 또는 참조하는 빈의 이름
- 생성자 파라미터 값 또는 참조:
  - DI에 사용할 생성자 파라미터 이름과 값 또는 참조할 빈의 이름
- 지연된 로딩 여부, 우선 빈 여부, 오토와이어링 여부, 부모 빈 정보, 빈 팩토리 이름 등

스프링 컨테이너는 각 빈에 대한 정보를 담은 설정 메타정보를 읽어들인 뒤에, 이를 참고해서 빈 오브젝트를 생성하고 프로퍼티나 생성자를 통해 의존 오브젝트를 주입해주는 DI 작업을 수행한다.

빈 메타정보의 항목들은 대부분 default 값이 있지만 싱글톤으로 관리되는 빈 오브젝트를 등록할 때 반드시 제공해줘야 하는 정보는 빈 이름과 POJO 클래스 뿐이다.
```
// Hello 클래스 빈 등록

StaticApplicationContext ac = new StaticApplicationContext(); // IoC 컨테이너 생성. 생성과 동시에 컨테이너로 동작
ac.registerSingleton("hello", Hello.class); // Hello.class를 싱글톤 빈으로 컨테이너에 등록

// IoC 컨테이너가 등록한 빈을 생성했는지 확인하기 위해 빈을 요청하고 null인지 확인한다.
Hello hello = ac.getBean("hello", Hello.class);
assertThat(hello, is(notNullValue()));
```
`StaticApplicationContext`는 코드에 의해 설정 메타정보를 등록하는 기능을 제공하는 애플리케이션 컨텍스트이다. 테스트용으로 적당하다.  
IoC 컨테이너가 관리하는 빈은 오브젝트 단위이다. 보통 클래스당 하나의 오브젝트를 만드기는 하지만, 경우에 따라서 하나의 클래스를 다른 설정을 지정해두고 사용하기 위해 여러 개의 빈으로 등록하기도 한다.

**요약**  
→ `StaticApplicationContext`가 디폴트 메타정보를 사용해서 싱글톤 빈을 등록해주는 `registerSingleton()` 메서드를 사용.

<br>

이번에는 BeanDefinition 타입의 설정 메타정보를 직접 만들어 IoC 컨테이너에 등록하는 방법을 사용해보자.  
`BeanDefinition 인터페이스`의 가장 기본적인 구현 클래스인 `RootBeanDefinition`을 알아보자. `RootBeanDefinition 오브젝트`를 만들어서 빈에 대한 설정정보를 넣어주고 IoC 컨테이너에 등록할 수 있다.
```
// BeanDefinition을 이용한 빈 등록

// 빈 메타정보를 담은 오브젝트 생성
BeanDefinition hello1 = new RootBeanDefinition(Hello.class);

// 빈의 name property에 들어갈 값을 지정 → <property name="name" value="Spring" />
hello1.getPropertyValues().addPropertyValue("name", "Spring");

// 위에서 생성한 빈 메타정보를 hello2라는 이름을 가진 빈으로 등록한다 → <bean id="hello2" … />
ac.registerBeanDefinition("hello2", hello);
```
```
// 검증 코드

Hello hello2 = ac.getBean("hello2", Hello.class);

// BeanDefinition으로 등록된 빈이 컨테이너에 의해 만들어지고 프로퍼티 설정이 됐는지 확인
assertThat(hello2.sayHello(), is("Hello Spring"));

// 처음 등록한 빈과 두 번째 등록한 빈이 모두 동일한 Hello 클래스지만 별개의 오브젝트로 생성됨
assertThat(hello1, is(not(hello2)));

// 컨테이너에 등록된 빈 메타정보의 개수
assertThat(ac.getBeanFactory().getBeanDefinitionCount(), is(2));
```
이번에는 `Hello` 타입의 빈과 `StringPrinter` 타입의 빈을 `hello`와 `printer`라는 빈 이름으로 생성하고 `printer 빈`이 `hello 빈`에게 DI 되도록 만들고 이 두 개의 빈을 등록하고 DI한 후 두 개의 빈이 서로 연동해서 동작하는지 확인하는 테스트 메서드를 만들어보자.
```
@Test
void registerBeanWithDependency() {

  // IoC 컨테이너 생성
  StaticApplicationContext ac = new StaticApplicationContext();

  // StringPrinter를 "printer"라는 이름의 빈으로 등록
  ac.registerBeanDefinition("printer", newRootBeanDefinition(StringPrinter.class));

  // Hello 클래스의 빈 메타정보 생성
  BeanDefinition helloDef = new RootBeanDefinition(Hello.class);

  // 메타정보에 프로퍼티 값 채우기(일반 값 하나, 레퍼런스 값 하나)
  helloDef.getPropertyValues().addPropertyValue("name", "Spring"); // setName("Spring");
  helloDef.getPropertyValues().addPropertyValue("printer", new RuntimeBeanReference("printer")); // DI 설정

  // 완성된 Hello 메타정보를 "hello"라는 이름의 빈으로 등록
  ac.registerBeanDefinition("hello", helloDef);

  // 컨테이너에서 "hello"라는 빈을 꺼냄(컨테이너가 내부적으로 객체를 만들고 printer를 주입)
  Hello hello = ac.getBean("hello", Hello.class);

  // hello 오브젝트의 기능을 실행(내부적으로 주입받은 printer의 append()가 호출)
  hello.print();

  assertThat(ac.getBean("printer").toString(), is("Hello Spring");
}
```
IoC 컨테이너가 POJO 클래스와 설정 메타정보를 이용해 최종 사용할 애플리케이션 런타임 오브젝트를 어떻게 만들어내는지에 대한 코드를 보았다. 이렇게 애플리케이션을 구성하는 빈 오브젝트를 생성하는 것이 IoC 컨테이너의 핵심 기능이다.
>IoC 컨테이너는 일단 빈 오브젝트를 생성되고 관계가 만들어지면 그 뒤로는 거의 관여하지 않는다. 기본적으로 싱글톤 빈은 애플리케이션 컨텍스트의 초기화 작업 중에 모두 만들어진다.
