## IoC 컨테이너의 종류와 사용 방법

스프링이 제공하는 ApplicationContext 구현 클래스에는 어떤 종류가 있고 어떻게 사용되는지 살펴보자.

### StaticApplicationContext
- `StaticApplicationContext`는 코드를 통해 빈 메타정보를 등록하기 위해 사용한다. 
- 스프링의 기능에 대한 학습 테스트를 만들 때를 제외하면 실제로 사용되지 않는다.
- 스프링의 웹 관련 기능을 공부하고 학습테스트로 검증해보고 싶을 때 유용하게 쓸 수 있다.
  - 그때는 서브클래스인 `StaticWebApplicationContext`를 사용할 것이다.
> 이런 `StaticApplicationContext`는 실전에서는 사용하면 안된다.

<br>

### GenericApplicationContext
- 가장 일반적인 `ApplicationContext`의 구현 클래스이다.
- 실전에서 사용될 수 있는 모든 기능을 갖추고 있는 `ApplicationContext`이다.
- 컨테이너의 주요 기능을 DI를 통해 확장할 수 있도록 설계되어 있다.
- XML 파일과 같은 외부의 리소스에 있는 빈 설정 메타정보를 리더를 통해 읽어들여서 메타정보로 전환해서 사용한다.
- JUnit 테스트는 테스트 내에서 사용할 수 있도록 `GenericApplicationContext`를 자동으로 만들어준다.

<br>

`XML`로 작성된 빈 설정정보를 읽어서 컨테이너에게 전달하는 대표적인 빈 설정정보 리더는 `XmlBeanDefinitionReader`이다. 이 리더를 `GenericApplicationContext`가 이용하도록 해서 `hello 빈`과 `printer 빈`을 등록하고 사용하게 만들어보자.
```
<!-- XML로 만든 빈 설정 메타정보 -->

<?xml version="1.0" encoding="UTF-8"?>
<beans xmlns="http://www.springframework.org/schema/beans"
       xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance"
       xsi:schemaLocation="http://www.springframework.org/schema/beans
       http://www.springframework.org/schema/beans/spring-beans-3.0.xcsd">

    <bean id="hello" class="springbook.learningtest.spring.ioc.bean.Hello">
       <property name="name" value="Spring" />
       <property name="printer" ref="printer" />
    </bean>

    <bean id="printer"
          class="springbook.learningtest.spring.ioc.beran.StringPrinter" />
<beans>
```
```
@Test
void genericApplicationContext() {
  GenericApplicationContext ac = new GenericApplicationContext();

  XMLBeansDefinitionReader reader = new XMLBeansDefinitionReader(ac);

  // XMLBeansDefinitionReader는 클래스패스로 정의된 리소스로부터 파일을 읽는다.
  reader.loadBeanDefinitions(
    "springbook.learningtest/spring/ioc/genericApplicationContext.xml");

  ac.refresh(); // 모든 메타정보가 등록이 완료됐으니 애플리케이션 컨테이너를 초기화하는 명령이다.

  Hello hello = ac.getBean("hello", Hello.class);
  hello.print();

  assertThat(ac.getBean("printer").toString(), is("Hello Spring"));
}
```

스프링은 `XML` 말고도 `properties` 파일에서 빈 설정 메타정보를 가져오는 `PropertiesBeansDefinitionReader`도 제공한다.
```
printer.(Class)=springbook.learningtest.spring.ioc.bean.StringPrinter

hello.(Class)=springbook.learningtest.spring.ioc.bean.Hello
hello.name=Spring
hello.printer(ref)=printer
```

`GenericApplicationContext`는 빈 설정 리더를 여러 개 사용해서 여러 리소스로부터 설정 메타정보를 읽어들이게도 할 수 있다. 모든 설정 메타정보를 가져온 후에 `refresh()` 메서드를 한번 호출해서 `ApplicationContext`가 필요한 초기화 작업을 수행하게 해주면 된다.

```
// 스프링 컨텍스트 테스트

@RunWith(SpringJUnit4ClassRunner.class)
@ContextConfiguration(locations = "/test-applicationContext.xml") // ApplicationContext 생성과 XML 파일을 읽어오고 초기화까지 수행
public class UserServiceTest {
  @AutoWired ApplicationContext ac;
}
```

### GenericXmlApplicationContext
`GenericApplicationContext`와 `XmlBeansDefinitionReader`를 합친게 `GenericXmlApplicationContext`이다.  
- `GenericXmlApplicationContext`는 `XML` 파일을 읽어들이고 `refresh()`를 통해 초기화하는 것까지 한줄로 끝낼 수 있다.
- `XML`파일로 설정을 만들고 `ApplicationContext`에서 `XML`을 읽어서 사용하는 코드를 만들 필요가 있다면 사용하기에 적당하다.
- `XML` 파일정보는 리소스 로더가 읽을 수 있는 형식으로 `GenericXmlApplicationContext 생성자에 넣어주면 된다.
```
// GenericXmlApplicationContext의 사용 코드

GenericXmlApplicationContext ac = new GenericXmlApplicationContext(
    "springbook/learningtest/spring/ioc/genericApplicationContext.xml");

Hello hello = ac.getBean("hello", Hello.class);
...
```

### WebApplicationContext
- 스프링 애플리케이션에서 가장 많이 사용되는 `ApplicationContext`로 `ApplicationContext`를 확장한 인터페이스이다.  
- 이름 그대로 웹 환경에서 사용할 때 필요한 기능이 추가된 `ApplicationContext`이다.  
스프링 애플리케이션은 대부분 서블릿 기반의 독립 웹 애플리케이션<sup>WAR</sup>으로 만들어지기 때문이다.

<br>

`WebApplicationContext`의 사용 방법을 이해하려면 스프링의 IoC 컨테이너를 적용 했을 때 애플리케이션을 기동시키는 방법에 대해 살펴볼 필요가 있다. 스프링 IoC 컨테이너는 빈 설정 메타정보를 이용해 빈 오브젝트를 만들고 DI 작업을 수행한다. 하지만 그것만으로 애플리케이션이 동작하지 않는다. 마치 자바 애플리케이션의 `main()` 메서드처럼 어디에선가 특정 빈 오브젝트의 메서드를 호출함으로써 애플리케이션을 동작시켜야 한다.
```
// 스프링 애플리케이션의 기동 코드

ApplicationContext ac = ...

// IoC 컨테이너가 구성한 빈 오브젝트로 이루어진 애플리케이션을 기동할 오브젝트를 getBean()으로 가져온다.
Hello hello = ac.getBean("hello", Hello.class);
hello.print();
```

적어도 한 번은 IoC 컨테이너에서 요청해서 빈 오브젝트를 가져와야 한다. 이때는 `getBean()` 메서드를 사용해야 한다. 이후에는 다시 `getBean()`으로 빈 오브젝트를 가져올 필요가 없다. 빈 오브젝트끼리 DI로 서로 연결되어 있으므로 의존관계를 타고 필요한 오브젝트가 호출되면서 애플리케이션이 동작할 것이다.
```
# 웹 환경에서 스프링 애플리케이션이 기동하는 방식

[ 1단계: 웹 서버 가동 ] ──► [ 2단계: 웹용 컨테이너 생성 ] ──► [ 3단계: 메타정보 스캔 & DI ] ──► [ 4단계: 요청 대기 ]
 (톰캣 서블릿 컨테이너)        (ApplicationContext)       (BeanDefinition 기반 조립)       (서버 기동 완료)
```
- main() 메서드 안에서 애플리케이션 컨텍스트를 만든다.
- 설정 메타정보를 읽어들여서 초기화한 후에 애플리케이션 기동 책임을 맡은 POJO 빈을 요청한다.
- 메서드를 실행한다.
서블릿 컨테이너는 브라우저와 같은 클라이언트로부터 들어오는 요청을 받아서 서블릿을 동작시켜주는 일을 맡는다. 서블릿은 웹 애플리케이션이 시작될 때 미리 만들어둔 웹 애플리케이션 컨텍스트에게 빈 오브젝트로 구성된 애플리케이션의 기동 역할을 해줄 빈을 요청해서 받아둔다. 그리고 미리 지정된 메서드를 호출함으로써 스프링 컨테이너가 DI 방식으로 구성해둔 애플리케이션의 기능이 시작되는 것이다. 다행히도 스프링은 DispatcherServlet이라는 이름의 서블릿을 제공한다.
>**DispatcherServlet의 기능**  
웹 환경에서 애플리케이션 컨텍스트를 생성하고 설정 메타정보로 초기화해주고, 클라이언트로부터 들어오는 요청마다 적절한 빈을 찾는다.

**WebApplicationContext의 특징**  
Tomcat같은 ServletContainer가 관리하는 웹 환경으로부터 필요한 정보를 가져오거나 웹 환경에 스프링 컨테이너 자신을 노출할 수 있다.  

<br>

`WebApplicationContext`는 웹 서버(Tomcat)와 긴빌하게 연결되어 있어서, 웹 서버의 정보를 가져와 쓸 수 있고, 반대로 웹 서버 측 일발 객체(필터, 서블릿 등)들이 스프링 컨테이너 안에 있는 빈을 꺼내 쓸 수 있도록 징검다리 역할을 해준다.
